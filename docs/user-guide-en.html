ReplayKit User Guide Test Automation Platform — Record, Replay, and Verify in One

System Overview

ReplayKit is a test automation platform that records user interactions on Android and IVI (In-Vehicle Infotainment) devices, automatically replays them while verifying results through screenshot-based image comparison (SSIM), and generates Excel reports.

ComponentTechnologyRole
BackendPython (FastAPI)Device control, record/playback engine, image comparison
FrontendReact + TypeScript + Ant DesignUser interface (web browser)
Device ConnectionADB / TCP Socket / SerialAndroid, HKMC IVI, serial devices, etc.

Core Workflow

① Connect Device → ② Record Actions → ③ Set Expected Images → ④ Play Scenario → ⑤ Review Results & Report

Installation

Method 1: Using the Installer (Recommended)

  1. Run ReplayKit_Setup_1.0.exe → Allow administrator privileges
  2. Confirm the installation path (default: C:\ReplayKit)
  3. Select components:
    • ReplayKit Core — Base (required, includes embedded Python)
    • DLT Viewer SDK — Check if you use the DLT log viewer GUI (included in Standard by default)
    • Vimba X SDK — Check if you use a Vision Camera (included in Full selection)
  4. After file copying completes, the setup window (setup.bat) opens automatically
  5. Python is included in embedded form, so no separate installation is needed
  6. If Git is not installed — The Git installer launches automatically:
    • Click Next with the default settings to install
    • After installation, press any key to continue
  7. Python packages are installed automatically → Git repository initialized → Done
  8. Launch ReplayKit after installation completes
💡 Embedded Python The distribution includes Python 3.10 Embedded, so there is no need to install Python separately on your system. When setup.bat runs, it automatically installs pip and required packages into the embedded Python. In a development environment, you can also use system Python 3.10 + venv.

Running the App

Running the Distribution

Double-click ReplayKit.bat to launch the server management GUI.

Development: Server Management GUI

python server.py

A Tkinter-based GUI window opens, allowing you to manage the backend and frontend simultaneously. Once the server is ready, the browser opens automatically.

Development: Manual Launch

# Terminal 1: Backend
venv\Scripts\activate
python -m uvicorn backend.app.main:app --reload --port 8000

# Terminal 2: Frontend

After launching, navigate to http://localhost:5173 in your browser to see the ReplayKit interface.

💡 Tip The frontend automatically proxies API requests to the backend, so no separate CORS configuration is needed.

Updates

ReplayKit supports Git-based automatic updates. Click the Sync + Start button in the server management GUI to:

  1. Automatically reset local changes (to prevent build file conflicts)
  2. Download the latest code via Git (git pull)
  3. Automatically install updated Python packages
  4. Automatically restart the server and open the browser
💡 User Data Preserved During Updates Scenarios, test results, device settings, path configurations, and other user data are not tracked by Git, so they remain unchanged during updates.

Device Management

On the Device tab — ReplayKit's first screen — you register and manage the devices used for testing. Devices are divided into primary devices and auxiliary devices.

Primary Device

The primary device is the target for screen interactions and screenshot capture.

TypeConnectionDescription
ADB USB or WiFi Android device. Connect via USB or WiFi using IP:port.
Example: 192.168.1.100:5555
HKMC TCP Socket HKMC 6th generation IVI system. Connect via IP and TCP port (default 5000).
Screen type selectable: Front Center, Rear Left, Rear Right, Cluster
Vision Camera GigE Vision (UDP) GigE Vision industrial camera. Automatically detected via GVCP protocol during scanning, displaying MAC address, current IP, and subnet information. Additional features available when Vimba X SDK is installed.

Connecting a Device

  1. Click the Scan button — ADB, HKMC, and Vision Camera devices are detected simultaneously
  2. Select the desired device from the detected list and click Connect
  3. If a device is not detected, you can manually enter the address in the Manual Connect tab
⚠️ ADB Device Preparation Settings → Developer Options → USB Debugging must be enabled on the Android device. For WiFi connection, the adb tcpip 5555 command must be executed first.

Vision Camera Connection (GigE Vision)

GigE Vision cameras connected to the network are automatically detected during scanning. The detection results show both the camera's current IP and the PC's network interface IP.

  1. When a camera is detected during Scan, its MAC, model, and current IP are displayed
  2. If the camera IP is on a different subnet than the PC, click the Change IP button
  3. In the ForceIP dialog, an IP matching the PC's subnet is automatically suggested
  4. After confirming, the camera IP is changed and a rescan is performed automatically
  5. Now click the Connect button — the values are populated in the Manual Connect tab, and click Connect to finish
💡 What is ForceIP? ForceIP is a forced IP assignment feature defined in the GigE Vision standard. When a camera is on a different subnet from the PC and cannot communicate, it changes the camera's IP to the same subnet as the PC to enable connection. This setting may be reset when the camera reboots.

Auxiliary Devices

Auxiliary devices support the primary device. They are used for serial communication, module control, and more.

TypeConnectionDescription
Serial COM Port / USB Serial communication device. Baudrate configurable (default 115200)
Module Plugin Extension modules (CAN communication, power control, etc.). Dynamically loaded plugins
Bench UDP Bench equipment. Connect via IP:UDP port

Device Actions

  • Rename — Edit the display name of a device
  • Edit Info — Change address, baudrate, module assignment, etc.
  • Disconnect — Unregister a device
  • Restart ADB Server — Restart the ADB server when connection issues occur
  • Reconnect — Attempt to reconnect a disconnected device

Recording

In the Recording tab, you watch the device's live screen and record user interactions to create test scenarios. Each interaction is saved as a step.

Start Recording

  1. Enter a scenario name (e.g., "Settings_WiFi_Connection_Test")
  2. Optionally add a description
  3. Select a target device (from the primary device list)
  4. Click the Start Recording button
  5. The device screen streams in real time — click/drag on the screen to record interactions
  6. When all interactions are done, click Stop Recording to save the scenario
💡 Continue Recording You can load an already-saved scenario and add steps at the end. Load Saved Scenario → Select scenario → Continue Recording

Adding Steps Manually

Use the Add Step area at the bottom of the screen during recording or in edit mode. Select the step type, enter the required values, and click the Add button.

Expected Images

To verify that each step executed correctly, set an expected image. During playback, the actual screenshot is compared with the expected image to determine Pass/Fail.

  1. Find the step you want to set an expected image for in the step list
  2. Click the 📷 button to automatically capture the current device screen as the full-screen expected image
  3. The captured image appears as a thumbnail next to the step
  4. If needed, click the image to enlarge it or change the comparison mode
✅ Auto-Capture Feature When a step without an expected image is executed, the resulting screenshot is automatically saved as the expected image. This is convenient when creating a scenario for the first time.

Image Comparison Modes

Four modes are available for comparing expected images with actual screenshots:

ModeIconDescriptionUse Case
Full Screen FULL Compare the entire screen using SSIM When the entire screen must match (default)
Single Crop SINGLE_CROP Crop a specific area and use template matching When checking only specific UI elements like status bar or notifications
Exclude Region FULL_EXCLUDE SSIM comparison excluding specified regions When ignoring changing areas like clock or battery
Multi Crop MULTI_CROP Compare multiple regions individually (all must pass) When checking multiple UI elements simultaneously

Editing Comparison Regions

  • Edit Exclude Regions (scissors icon) — Drag on the expected image to designate regions to exclude (shown in red)
  • Edit Crop Regions — Drag to designate comparison regions (shown in green)
  • ROI Settings — Designate only the target area for comparison in Full Screen mode
  • Each region can be modified by clicking it in the list or deleted with the X button
💡 Similarity Threshold The default similarity threshold is 0.95 (95%). Scores at or above this value result in Pass; below it results in Fail. In Multi Crop mode, individual thresholds can be set for each crop.

Step Editing

Recorded steps can be modified at any time. From each step's menu (⋯):

  • Edit Command — Change parameters such as coordinates, text, or commands
  • Retake Expected Image — Replace the expected image with the current device screen
  • Test This Step — Execute and verify only this step to check the result
  • Insert Wait — Insert a wait time after this step
  • Delete — Remove the step

For steps that require coordinate changes (tap, long_press, swipe), clicking/dragging on the device screen in edit mode automatically applies the new coordinates.

Conditional Jump

You can specify the next step to execute based on the verification result (Pass/Fail) of a step. This allows you to branch the test flow or implement logic such as skipping on failure.

SettingDescription
Jump on PassJump to the specified step on verification success (default: next step)
Jump on FailJump to the specified step on verification failure, or terminate (END)
⚠️ Infinite Loop Warning When setting conditional jumps, be careful to avoid circular references that loop back to a previous step.

Scenario Management

In the Scenario tab, you can view, execute, and manage recorded scenarios.

Scenario List

All saved scenarios are displayed in a list. Available actions for each scenario:

  • Play — Execute the scenario on a device
  • Copy — Duplicate with a different name
  • Rename — Edit the scenario name
  • Merge — Combine multiple scenarios into one
  • Delete — Remove the scenario (confirmation required)

Playback (Execution)

Execute a scenario to automatically control the device and verify results.

  1. Click the Play button next to the scenario
  2. Confirm device mapping — Map the device the scenario was recorded on to the currently connected device. You can change the mapping here if the devices differ
  3. Set the repeat count if needed (default: 1)
  4. Click Execute to start playback
  5. Results are displayed in real time as each step executes:
    • Pass — Similarity at or above threshold
    • Warning — Borderline level
    • Fail — Similarity below threshold
    • Error — Execution error occurred

Playback Controls

ButtonFunction
PauseWait after the current step completes
ResumeContinue paused playback
StopImmediately stop playback and save results up to the current point

Group Management

Multiple scenarios can be grouped together for sequential execution. When executing as a group, conditional jumps between scenarios are also possible.

  1. Create Group — Enter a group name and create it
  2. Add scenarios to the group (multiple allowed)
  3. Reorder scenarios by dragging
  4. Set conditional jumps between scenarios if needed:
    • Scenario A Pass → Jump to Scenario C
    • Scenario A Fail → Terminate (END)
  5. Click Group Play to execute the included scenarios sequentially

Export / Import

Export

Package scenarios and groups as ZIP files for distribution to other PCs.

  1. Select the scenarios/groups to export
  2. Click Export → A ZIP file is generated
  3. If a save path is configured in settings, the file is saved there; otherwise it downloads via the browser

Import

  1. Click Import → Select a ZIP file
  2. In the preview screen, review the included scenario list and check for conflicts
  3. Choose how to handle conflicts:
    • Skip — Keep the existing scenario
    • Overwrite — Replace with the imported scenario
    • Rename — Add with a new name
  4. Click Apply to complete the import

Result List

  • Total steps / Pass / Fail count

You can filter by scenario name or bulk delete selected entries.

Tips & Troubleshooting

FAQ

Q: My device doesn't appear in the scan

Check the following:
  • Android: Verify that USB debugging is enabled
  • Android: Reconnect the USB cable and approve the authorization popup on the PC
  • HKMC: Verify they are on the same network and that the firewall is not blocking
  • Serial: Verify the driver is installed and no other program is occupying the COM port
  • Try Restart ADB Server

Q: Similarity scores are low during playback

  • If there are time-varying areas (clock, battery, etc.), use Exclude Region mode
  • If you only want to check specific elements rather than the full screen, use Single Crop or Multi Crop
  • You can adjust the similarity threshold (default 0.95)
  • Differences may occur if the device resolution differs from when the scenario was recorded

Q: I want to run a scenario on a different PC

  • Export the scenario as a ZIP file
  • Import the ZIP file on the other PC
  • During playback, select the current PC's device in the device mapping step

Q: The server connection was lost

  • A "Connecting to server..." message appears at the top of the screen
  • Auto-reconnect is attempted, and the connection recovers automatically once the server is running again
  • If unresolved, restart server.py

Q: I want to adjust the wait time between steps

Change the delay_after_ms value in each step's Edit Command. The default is 1000ms (1 second); increase it if screen transitions are slow.

Module Guide

Modules are extensions that can be invoked from scenarios via module_command steps. They can be added as auxiliary devices on the Device page, or inserted directly as steps during scenario editing.

CMD — Command Execution Module

A module that executes Windows system commands. Used for launching apps, managing processes, and checking system status during tests.

Connection Settings

No separate connection setup is needed. Simply select CMD when choosing a module and it's ready to use.

Available Functions

FunctionDescriptionParametersReturn Value
Run Execute a command and wait for completion (blocking) command — Command to execute
timeout — Max wait time in seconds (default 300) 		Execution output (stdout + stderr)
RunBackground Execute in the background (non-blocking) command — Command to execute Process ID in the form PID:12345
Kill Terminate a background process pid — PID of the process to terminate Termination result message
ListBackground List running background processes (none) List of alive PIDs (comma-separated)

Usage in Scenarios

Difference by step type:
  • cmd_send — Executes the command and ignores the result. Use for cases where only execution is needed, such as launching apps or copying files
  • cmd_check — Executes the command and compares the output against an expected value. Set expected value and match_type (contains / exact / regex)
  • module_command — Directly calls CMD module functions. Use for advanced features like RunBackground or Kill

Usage Examples

PurposeStep TypeConfiguration
Launch app cmd_send command: start notepad.exe
Ping verification cmd_check command: ping -n 1 192.168.1.1
expected: TTL=, match_type: contains
Background execution module_command module: CMD, function: RunBackground
args: command=python long_task.py
Terminate process module_command module: CMD, function: Kill
args: pid=12345

Usage in Scenarios

All functions are called via module_command steps. The most common use case is DLT log keyword verification:

Usage Examples

PurposeFunctionConfiguration
Wait for log keyword (blocking) WaitLog keyword: BOOT_COMPLETE, timeout: 60
→ PASS if keyword appears within 60 seconds; pauses step execution
AND condition verification WaitLog keyword: APP_START navigation
→ Searches for a line containing both words
Background monitoring StartMonitor keyword: ERROR, timeout: 120
→ Next step proceeds immediately; monitors in a separate thread for 120 seconds
Check monitoring result GetMonitorResult monitor_id: mon_1
→ Returns PASS/FAIL/RUNNING
Clear logs ClearLogs → Clears previous logs to set a new verification starting point
Visualize logs with GUI LaunchViewer project_file: CCIC_DLT_VIEWER_PRJ.dlp
WaitLog vs StartMonitor:
  • WaitLog — Blocking: Pauses step execution until the keyword appears. Use for definitive verification.
  • StartMonitor — Non-blocking: Proceeds to the next step immediately while monitoring in the background. Check later with GetMonitorResult.
  • In the step add UI, select WaitLog and check "Background monitoring" to automatically switch to StartMonitor.
Typical DLT Verification Scenario Flow:
  1. Add DLTViewer module as an auxiliary device (set ECU IP)
  2. Log receiving starts automatically on connection
  3. Clear buffer with ClearLogs
  4. Perform test actions (screen touch, app launch, etc.)
  5. Verify with WaitLog that the expected DLT log appears
